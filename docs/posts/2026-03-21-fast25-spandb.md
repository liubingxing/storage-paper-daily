# 【论文精读】SpanDB: A Fast, Cost-Effective LSM-tree Based KV Store on Hybrid Storage

> **会议**: FAST'25 | **日期**: 2026-03-21
> **标签**: key-value store, LSM-tree, hybrid storage

# 深度技术分析：SpanDB - A Fast, Cost-Effective LSM-tree Based KV Store on Hybrid Storage

---

## 论文基本信息

- **会议**: FAST'25 (USENIX Conference on File and Storage Technologies)
- **年份**: 2025
- **研究方向**: 存储系统，重点关注基于 LSM-tree 的键值存储 (Key-Value Store) 在混合存储 (Hybrid Storage) 环境下的高性能与低成本设计。

---

## 研究背景与动机

### 研究问题及其表现
传统的 LSM-tree（Log-Structured Merge Tree）是现代键值存储系统的核心数据结构，因其良好的写性能、可扩展性和简单性，被广泛应用于分布式存储系统（如 LevelDB、RocksDB 和 Cassandra）。然而，当 LSM-tree 部署在**混合存储环境（Hybrid Storage）**中时，面临以下问题：

1. **写放大的挑战（Write Amplification）**：
   - LSM-tree 通过分层结构将数据逐层合并，导致写放大问题严重。
   - 在混合存储中，写放大会加速 SSD 的寿命消耗，增加运维成本，同时拖慢写入性能。

2. **读性能受限**：
   - LSM-tree 的读性能受制于多层次结构（多次查找）以及随机读访问模式，尤其在低延迟、高并发的工作负载下，读性能成为瓶颈。

3. **混合存储利用率低**：
   - 混合存储（如 SSD+HDD 或 NVMe+QLC SSD）环境中，如何高效利用不同存储介质的性能与成本优势是一个难点。
   - 现有的存储系统无法充分利用快速存储（如 NVMe SSD）和廉价存储（如 QLC SSD 或 HDD）之间的特性差异。

### 问题重要性
上述问题对实际系统有以下影响：
1. **成本**：写放大增加了硬件更换频率，特别是对寿命有限的 SSD。
2. **性能**：混合存储的异构特性使得资源利用率不均衡，导致性能无法充分释放。
3. **扩展性**：简单地堆叠硬件资源并不能解决写放大和读性能问题，影响系统扩展能力。

### 现有方案及不足
1. **传统 LSM-tree（如 RocksDB）**:
   - 优化主要集中在单一存储介质（如 NVMe SSD）上，对混合存储支持不足。
   - 写放大问题依然严重，尤其在多层次合并（compaction）过程中。

2. **混合存储系统（如 PebblesDB）**:
   - 尽管引入了部分优化（如分区合并），但未能充分利用存储介质的异构特点。

3. **基于缓存的优化**:
   - 通过引入 DRAM 缓存或 NVMe 缓存改善读性能，但无法从根本上解决写放大的系统性问题。

### 核心 Insight
SpanDB 的核心 Insight 是：
- **分层存储优化**：通过对 LSM-tree 的层次结构进行重新设计，充分利用混合存储的异构特性，将性能关键的操作优先分配到快速存储介质，将容量密集型操作分配到低成本存储介质。
- **异构感知的 compaction 策略**：动态调整 compaction 策略以减少写放大，同时保证高效的读取路径。
- **适配多种存储场景**：在性能和成本之间动态权衡，提供一种通用的混合存储解决方案。

## 架构设计图

以下是 SpanDB 的核心架构图，展示其主要组件及数据流设计：

```mermaid
flowchart TB
    subgraph 高速存储介质[快速存储 (NVMe SSD)]
        L0[MemTable (内存)] -->|Flush| L1[高速层 (Compact 层)]
    end
    subgraph 容量存储介质[高容量存储 (QLC SSD/HDD)]
        L2[中间层] -->|Compact| L3[容量层]
    end
    UserApp[用户查询/写入] -->|Put/Get| L0
    L1 -->|Compaction| L2
    L2 -->|Compaction| L3
    UserApp -->|Point Lookup| L1
    UserApp -->|Range Query| L2
    subgraph Metadata[元数据管理]
        MetaIndex[元数据索引]
        HotDataFilter[热点数据过滤器]
    end
    MetadataSync[元数据同步] --> L1
    MetadataSync --> MetaIndex
    MetaIndex --> L0
```

**图解说明**：
1. **快速存储层**: 包含 `MemTable` 和高速存储层（L1），用于处理写入请求及热点数据。
2. **容量存储层**: 包含中间层（L2）和容量层（L3），用于存储冷数据，优化存储成本。
3. **元数据管理**: 包括元数据索引和热点数据过滤器，用于跟踪数据分布与访问特性。
4. **数据流**: 写入数据从 `MemTable` 开始，依次进行 `Flush` 和跨层 `Compaction`，最后存储到容量层。
5. **控制流**: 元数据同步管理数据分布信息，支持快速查询和动态策略调整。

## 核心设计与技术贡献

### 整体架构
SpanDB 的架构基于改进的 LSM-tree，结合混合存储的异构特性，分为以下核心组件：
- **MemTable**:
  - 用于接收用户的写入请求，暂存数据，避免频繁写入底层存储。
- **高速层（L1）**:
  - 部署在快速存储介质（如 NVMe SSD）上，存储热点数据，并进行部分 compaction 操作。
- **中间层（L2）**:
  - 作为过渡层，跨越快速存储和容量存储，存放冷数据。
- **容量层（L3）**:
  - 部署在廉价的容量存储介质上（QLC SSD 或 HDD），存储冷数据，提供高性价比的存储能力。

### 关键技术点

#### 技术点 1: 异构感知的 LSM-tree 层次划分
1. **要解决的子问题**:
   - 传统 LSM-tree 的层次划分固定，未考虑存储介质的异构特性，导致性能和成本无法兼顾。
2. **设计方案**:
   - 引入基于存储介质的层次划分，将 LSM-tree 分为高速层和容量层。
   - 高速层（L1）优先处理写入和热点数据，利用快速存储的低延迟特性。
   - 容量层（L2/L3）用于存放写频率较低或冷数据，利用容量存储的低成本优势。
3. **设计权衡**:
   - 优化写放大的同时，可能引入额外的元数据管理开销。
4. **与现有技术的区别**:
   - 与 RocksDB 相比，引入了存储介质感知的层次划分，动态调整数据分布。
   
#### 技术点 2: 异构感知的 Compaction 策略
1. **要解决的子问题**:
   - LSM-tree 中的 compaction 操作代价高昂，导致写放大严重。
2. **设计方案**:
   - 在高速层优先进行小范围的增量 compaction，避免频繁的大规模合并。
   - 在容量层启用延迟 compaction 策略，减少对容量存储的写入压力。
3. **设计权衡**:
   - 高速层的 compaction 操作可能在高负载下成为瓶颈，但通过动态调整可以缓解。
4. **与现有技术的区别**:
   - 通过感知存储介质性能，动态调整 compaction 策略，而非固定的分层合并。

#### 技术点 3: 热点数据过滤器
1. **要解决的子问题**:
   - 热点数据的频繁访问对容量层性能造成了影响。
2. **设计方案**:
   - 利用元数据索引和热点数据过滤器，将热点数据动态迁移至高速层。
3. **设计权衡**:
   - 引入了额外的元数据存储和管理开销，但换取了显著的读性能提升。
4. **与现有技术的区别**:
   - 热点数据感知能力显著增强，相比 RocksDB 显著提高了读性能。

### 创新点总结
SpanDB 的核心创新在于将传统 LSM-tree 与异构存储介质特性深度集成，通过分层存储优化、动态调整 compaction 策略以及热点数据迁移，显著提升了混合存储环境下的性能与成本效率。这种设计的独特性在于其充分利用了存储介质的异构性，而不是试图用单一优化策略解决所有问题。

## 实验评估亮点

### 实验环境和基准
- **硬件环境**:
  - 快速存储: NVMe SSD
  - 容量存储: QLC SSD 和 HDD
- **工作负载**:
  - 使用 YCSB（Yahoo! Cloud Serving Benchmark）模拟真实的读写负载。
- **对比基线**:
  - RocksDB, PebblesDB, LevelDB。

### 关键性能数据
1. 写放大减少 30%-50%。
2. 读性能提升 2x-3x，尤其在高并发场景下。
3. 存储成本降低 20%-30%（通过优化容量层存储利用率）。

### 实验结论
实验结果表明，SpanDB 在混合存储环境下提供了显著的性能与成本优势。其设计能够有效平衡写入性能、读取延迟和存储成本。

## 与工业界的关联

### 类似实践
- **RocksDB**:
  - Facebook 的 RocksDB 是目前工业界主流的 LSM-tree 实现，但其主要针对单一存储介质优化，缺乏对混合存储的支持。
- **Amazon Aurora**:
  - Amazon Aurora 的存储层也采用多级存储设计，启发了 SpanDB 的层次架构。
- **Ceph**:
  - Ceph 的分层存储和异构感知策略与 SpanDB 的设计有相似之处。

### 工程落地的可行性
SpanDB 的设计可以被工业界存储系统采用，但需要解决以下挑战：
1. **元数据管理复杂性**。
2. **快速存储资源分配**: 如何动态分配快速存储资源以支持不同工作负载的需求。
3. **容错性**: 数据在多种存储介质间迁移时，如何保证数据一致性和可靠性。

## 个人思考启发

### 值得学习的点
1. **异构感知的设计**：SpanDB 的设计充分考虑了存储介质的性能与成本差异，这种思想值得借鉴。
2. **动态优化能力**：通过动态调整 compaction 策略和数据分布，SpanDB 在性能与成本之间找到了很好的平衡。

### 潜在局限性
1. **复杂性成本**：分层设计和动态优化引入了额外的元数据开销和实现复杂度。
2. **局限性场景**：如果存储介质的性能差异不大（如全 NVMe 环境），其优势可能会减弱。

### 对存储系统从业者的启示
这篇论文提醒我们，在面对多样化的存储硬件时，设计系统时需要充分考虑硬件特性，避免一刀切的优化策略。此外，动态适配和异构感知将是未来存储系统设计的重要方向。
