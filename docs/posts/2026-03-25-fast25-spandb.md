# 【论文精读】SpanDB: A Fast, Cost-Effective LSM-tree Based KV Store on Hybrid Storage

> **会议**: FAST'25 | **日期**: 2026-03-25
> **标签**: key-value store, LSM-tree, hybrid storage

## 论文基本信息

- **论文标题**: SpanDB: A Fast, Cost-Effective LSM-tree Based KV Store on Hybrid Storage  
- **发表会议**: FAST 2025  
- **研究方向**: 分布式存储系统、LSM-tree（Log-Structured Merge Tree）优化、混合存储（Hybrid Storage）架构  

这篇论文聚焦于基于 LSM-tree 的 Key-Value Store（KV Store）在混合存储介质（如 NVMe SSD 和 HDD）上的性能和成本优化。论文提出了一种名为 **SpanDB** 的新型存储系统，旨在通过高效的架构和数据管理策略，突破现有 LSM-tree 系统在混合存储中性能与成本的限制。

---

## 研究背景与动机

### 背景

基于 LSM-tree 的 Key-Value Store（如 RocksDB、LevelDB）被广泛应用于数据库、分布式存储系统等领域，因其在写入密集型工作负载中的优越表现而备受青睐。然而，随着数据规模的爆发式增长，如何在成本和性能之间取得平衡成为一个核心挑战。在此背景下，混合存储架构（Hybrid Storage）逐渐成为一种吸引人的选择。

混合存储架构通过同时使用高性能的存储介质（如 NVMe SSD）和低成本、大容量的存储介质（如 HDD），试图平衡性能与成本。然而，现有的 LSM-tree 系统并未充分优化以利用混合存储的特点，导致以下问题：

### 问题描述

1. **写放大效应（Write Amplification）**:  
   LSM-tree 的 compaction 策略会导致大量重复写操作，尤其是低速 HDD 对写操作十分敏感，导致性能瓶颈和硬件磨损。  

2. **读放大效应（Read Amplification）**:  
   LSM-tree 的分层结构使得读取数据时可能需要在多个层级中查找，特别是跨 SSD 和 HDD 的数据访问会进一步放大延迟。

3. **空间放大效应（Space Amplification）**:  
   数据多次重写会使存储利用率降低，混合存储环境中的硬件成本压力进一步加剧。

4. **冷热数据管理不充分**:  
   现有方案缺乏对冷热数据的精细化管理，导致高性能的存储介质（如 SSD）未被充分利用。

### 为什么重要？

- **成本控制**: 随着数据规模快速增长，单纯依赖 NVMe SSD 的存储系统会带来高昂的成本，而混合存储可大幅降低总体拥有成本（TCO）。  
- **性能需求**: 混合存储的性能权衡问题直接影响了存储系统在大规模写入和复杂查询场景下的表现，尤其是在企业级应用中。  
- **存储寿命**: 降低写放大效应不仅能提升系统性能，还能显著延长存储设备的使用寿命。

### 现有方案与不足

1. **传统 LSM-tree 系统（如 RocksDB、LevelDB）**:  
   - 优点：成熟稳定，通用性强，适用于多种场景。  
   - 缺点：未针对混合存储设计，写放大和读放大问题严重，冷热数据管理粗放，缺乏存储层级感知。  

2. **基于缓存的优化方案（如 WiscKey、PebblesDB）**:  
   - 优点：通过分离 key 和 value 或调整 compaction 策略，降低了写放大。  
   - 缺点：仍未考虑混合存储的异构特性，对 HDD 的随机写性能影响较大。  

3. **混合存储专用优化方案（如 SlimDB, H-Store）**:  
   - 优点：利用混合存储的特点，优化冷热数据的分层存储。  
   - 缺点：设计复杂，往往需要依赖硬件特性，通用性较差。  

### 核心 Insight

SpanDB 的核心 Insight 是：通过重新设计 LSM-tree 的数据组织方式和 compaction 策略，以及引入存储层级感知的冷热数据放置机制，可以最大化利用混合存储的性能优势，同时显著降低存储成本。

## 核心设计与技术贡献

### 整体架构

SpanDB 的架构主要由以下几个核心组件构成：

1. **MemTable 和 Immutable MemTable**:  
   用于暂存写入数据，支持快速写入。数据会被批量刷新（flush）到存储介质中。  

2. **混合存储 LSM-tree**:  
   LSM-tree 被重新设计以适配混合存储架构，包括 SSD 层和 HDD 层。数据从上到下逐步降级存储，冷热数据分层放置。  

3. **查询路径优化模块**:  
   通过存储元数据索引和访问模式分析，优化读取路径，减少跨层读取的开销。  

### 关键技术点

#### 1. **层级感知的冷热数据放置策略**
- **要解决的子问题**:  
   在混合存储环境中，如何确保热数据优先留在高性能存储介质（如 SSD），而冷数据尽可能迁移到低成本介质（如 HDD），以提高性能并降低成本？

- **设计方案**:  
   SpanDB 引入了一种基于访问频率的冷/热数据分类机制，结合 LSM-tree 的层级结构：
   - 热数据优先保存在高层（SSD 层）。
   - 冷数据在 compaction 时优先下沉到低层（HDD 层）。
   - 使用动态阈值调整冷热数据的分界点，避免固定策略导致的分类不准确。

- **设计权衡**:  
   - **优点**: 提升了 SSD 的利用效率，同时减少了 HDD 的随机 I/O 压力。  
   - **缺点**: 需要额外的元数据存储和访问频率统计，增加了一定的系统复杂性。  

- **与现有技术的区别**:  
   - 区别于传统固定策略（如按时间戳简单划分热/冷数据），SpanDB 的动态分类更加精准，减少了误判带来的性能损失。

#### 2. **分层 compaction 优化**
- **要解决的子问题**:  
   如何减少 LSM-tree 在混合存储上的写放大效应，同时保证 compaction 的效率？

- **设计方案**:  
   SpanDB 提出了 **Span-compaction** 策略，核心思想是通过限制跨层 compaction 的频率和数据量，减少写放大。  
   - 在 SSD 层，优先执行小范围的局部 compaction，避免频繁写入 HDD。  
   - 在 HDD 层，采用大批量的归并排序（bulk merge），减少磁盘的随机写操作。  

- **设计权衡**:  
   - **优点**: 显著降低了写放大效应，减少了存储设备的磨损。  
   - **缺点**: 可能导致一定程度的延迟累积，需要通过调度优化缓解。

- **与现有技术的区别**:  
   - 传统 LSM-tree 的 compaction 策略是全局性的，未考虑存储层级差异，而 SpanDB 的 Span-compaction 是层级感知的。

#### 3. **查询路径优化**
- **要解决的子问题**:  
   如何减少跨层查询的延迟，尤其是在数据分布在 SSD 和 HDD 之间时？

- **设计方案**:  
   SpanDB 在元数据索引中引入了层级感知信息，结合基于访问模式的预测算法，优化查询路径：  
   - 如果数据可能在 SSD 中，优先访问高层。  
   - 避免不必要的 HDD 扫描，尽可能减少跨层操作。

- **设计权衡**:  
   - **优点**: 读取性能显著提升，尤其是小范围查询的延迟。  
   - **缺点**: 增加了索引维护的开销。  

- **与现有技术的区别**:  
   - 传统方案通常依赖简单的逐层查询，而 SpanDB 的查询路径优化更具智能化和层级感知。

### 创新点总结

- **动态冷热数据分类**：打破了传统静态冷热数据分类的局限性。  
- **层级感知的 compaction 策略**：显著降低写入和空间放大效应。  
- **智能查询路径优化**：通过层级感知和模式预测，实现了低延迟的混合存储访问。

## 实验评估亮点

### 实验环境和基准

- 环境：混合存储设备（NVMe SSD + HDD），16核服务器，128GB 内存。  
- 基准：YCSB（Yahoo! Cloud Serving Benchmark），评估了多种读写工作负载（A-F 模式）。  

### 对比系统

- **RocksDB**: 传统 LSM-tree 系统的代表。  
- **PebblesDB**: 通过调整 LSM-tree 的 compaction 优化写放大的系统。  
- **SlimDB**: 针对混合存储优化的 LSM-tree 系统。

### 关键性能数据

- 在混合读写负载下，SpanDB 的吞吐量相比 RocksDB 提升了 **2.5 倍**，相比 PebblesDB 提升了 **1.8 倍**。  
- 写放大效应降低了 **60%**，存储介质的耐用性大幅提升。  
- 在随机读取场景下，延迟降低了 **40%**，显著提升了用户体验。  

### 实验结论

- SpanDB 的冷热数据分层和 compaction 优化机制对混合存储环境非常有效。  
- 通过减少写放大和读放大，SpanDB 实现了更高的性能和更低的 TCO。

## 与工业界的关联

### 类似实践

- **Facebook 的 RocksDB**: 虽然是传统的 LSM-tree 系统，但正在探索混合存储相关优化，例如 blob 存储和分层存储。  
- **Amazon Aurora**: 使用分层存储架构，将冷数据迁移到更低成本的存储。  
- **Ceph 和 MinIO**: 在对象存储中已尝试将冷热数据分层。  

### 借鉴潜力

SpanDB 的层级感知设计和 compaction 优化策略，可以为工业界的存储系统提供指导，特别是分布式 KV 存储（如 TiKV、HBase）的优化方向。

### 工程挑战

- 动态数据分类的实时性和准确性对系统性能提出较高要求。  
- 系统复杂度增加，可能需要额外的计算资源和工程投入。

## 个人思考启发

1. **值得学习的点**:  
   - 论文展示了如何结合硬件特性和数据模型优化存储系统，尤其是混合存储架构的细粒度设计。  
   - 对 LSM-tree 的 compaction 和查询路径的优化思路非常值得借鉴。

2. **潜在局限性**:  
   - 对数据访问模式的依赖较强，若工作负载变化剧烈，可能导致优化效果不明显。  
   - 复杂的索引和数据分类机制可能带来额外的维护成本。

3. **启示**:  
   - 存储系统的设计应更关注硬件特性与实际工作负载的匹配。  
   - 在混合存储环境中，动态适应性（adaptive optimization）将成为未来的关键研究方向。  
   - 工程实践中，需权衡性能、成本和复杂性，选择适合的技术方案。
